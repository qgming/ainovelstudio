# 神笔写作 v0.5.0 更新日志

发布日期：2026-06-01

## 版本亮点

本次更新是 `0.4.0 -> 0.5.0` 的一次核心工程重构，主线是**将书籍项目重构为 Markdown 体系**。项目资料、设定、状态、规则、技能参考和 AI 可读上下文进一步回到可读、可编辑、可版本化的 Markdown 文件中，减少隐藏结构和专用 JSON 对作者工作流的干扰。

围绕这条主线，0.5.0 同时完成了 JSON 相关内容与工具的移除、上下文组织方式优化，以及写作技能体系的优化和扩展。目标是让 Agent 更稳定地读取项目事实源，让作者更容易检查、修改、备份和迁移自己的小说工程。

## 重点更新

### 1. 项目体系重构为 Markdown 优先

- 书籍项目的长期资料进一步统一到 Markdown 文件承载，降低专用结构对创作流程的侵入
- 项目说明、创作规则、人物设定、世界观、时间线、伏笔、章节状态和写作记忆更适合以 `.md` 文件沉淀
- Agent 读取上下文时优先面向人类可读的项目文件，而不是依赖难以维护的结构化 JSON 片段
- 作者可以直接在编辑器里查看和修改项目资料，减少“数据存在但看不见、改不动”的情况
- Markdown 文件天然适合备份、迁移、版本管理和跨工具复用，也更利于长期项目维护

### 2. 移除 JSON 相关内容与工具

- 清理旧版围绕 JSON 项目结构设计的内容、提示和工具入口
- 移除不再符合 Markdown 工作流的 JSON 读写与维护路径
- 降低 Agent 在 JSON schema、伪 JSON、结构化片段和正文文件之间来回转换的成本
- 减少模型因为格式修复、字段补全或 JSON 容错而消耗的无效上下文
- 创作资料的维护方式从“填结构”转向“写文档”，更符合小说作者实际整理设定的习惯

### 3. 上下文组织进一步优化

- 优化 Agent 运行时上下文，让模型更清楚当前书籍、当前文件、启用技能和可用事实源之间的关系
- 减少重复、过时或低价值的上下文注入，降低长会话中的噪音
- 强化“先读项目文件、再生成或修改内容”的协作方式，让正文、设定和状态文件成为主要依据
- 调整上下文表达，使 Markdown 项目资料更容易被模型检索、引用和回写
- 长篇写作场景下，项目事实源、技能规则、手动选择文件和工具结果的边界更清晰

### 4. 技能体系优化

- 优化内置技能的说明、触发边界和执行流程，使 Agent 更容易判断什么时候该读取哪个技能
- 技能内容进一步适配 Markdown 项目体系，强调读取项目文件、引用事实源和写回可维护文档
- 写作类技能更关注落地执行：先读资料、再计划、再写作或诊断、最后按需要回写项目文件
- 分析类和扫榜类技能更适合把结论沉淀为题材卡、拆文笔记、设定文档或项目记忆
- 质检类技能更强调诊断边界，先定位问题和证据，再决定是否改正文

### 5. 新增写作技能

本版本补充和扩展了面向小说生产链路的技能能力，覆盖更多创作场景：

- 正文落笔技法：视角、动作、对话、节奏、标点和人设味
- 作者风格提取与套用：把样本文本炼化为可复用风格基因
- 连续性审计：检查设定、伏笔、人设、时间线和能力边界
- 改编适配：支持长短篇互改、平台适配、旧稿重做和调性调整
- 书名简介包装：围绕标题、简介、标签和卖点优化作品门面
- 技能制作：把个人写作套路、流程或检查清单固化为本地技能

当前 README 已按 13 个内置技能重新整理：`story-long-write`、`story-long-analyze`、`story-long-scan`、`story-short-write`、`story-short-analyze`、`story-short-scan`、`story-prose-craft`、`story-deslop`、`story-author-style`、`story-continuity-audit`、`story-rewrite-adapt`、`story-title-blurb`、`skill-creator`。

### 6. README 与版本信息同步更新

- README 根据 Markdown 项目体系重新组织产品介绍、设计理念、快速上手和典型工作流
- 内置技能列表、核心功能、Agent 协作方式、数据隐私和开发者说明同步更新
- 项目版本元数据统一更新为 `0.5.0`
- 下载链接命名更新为 `ainovelstudio_0.5.0_windows_x64.exe` 与 `ainovelstudio_0.5.0_android_arm64.apk`

## 兼容与行为变化

- 0.5.0 之后，项目资料维护方式更偏向 Markdown 文档，不再鼓励使用旧 JSON 结构作为主要事实源
- 与 JSON 相关的旧工具和提示路径已移除或收敛，依赖旧 JSON 工作流的项目建议逐步迁移为 `.md` 文档
- Agent 上下文会更重视项目 Markdown 文件和明确选择的资料，输出也更倾向写回可读 Markdown
- 技能触发和技能内容经过调整，同一任务在 0.5.0 中可能会读取更匹配的技能或采用更清晰的执行流程
- 本次版本元数据更新不应改变已有书籍正文文件；建议升级前后都保留完整备份

## 建议迁移方式

如果你的旧项目里仍有 JSON 化设定或状态资料，建议逐步迁移为 Markdown：

1. 将人物、地点、势力、能力体系、时间线、伏笔等内容拆成 `.md` 文档。
2. 在 `.project/README.md` 写项目总览、当前进度和资料入口。
3. 在 `.project/AGENTS.md` 写本书硬规则、禁写项、风格约束和事实源优先级。
4. 将长期稳定设定放入 `.project/canon/`，将阶段性总结和风格经验放入 `.project/memory/`。
5. 让 Agent 后续任务先读取这些 Markdown 文件，再写作、分析或返修。
