# 状态更新规则

## 更新来源

状态更新优先来自：

- 新写出的正文
- 章节规划中的已确认推进
- `story-bible` 中已确认 canon
- `outline-manager` 中已确认结构

## 更新顺序

1. 先确认这次变化属于长期事实还是即时状态
2. 长期事实回 `story-bible`
3. 即时状态更新到 JSON
4. 如果结构推进变了，再同步 `outline-manager`

## 更新原则

- 初始化时优先从对应模板复制，再填当前真值
- 只写当前有效真值
- 不把整段剧情摘要硬塞进 JSON
- 不把多个概念混成一个字段
- 不删历史影响时，优先把状态改为关闭 / 已完成 / 已失效
- 每次更新后保持字段名一致

## 模板使用方式

- 新建状态文件时，优先直接复制模板
- 已有状态文件时，按模板补齐缺失字段
- 项目已经有稳定字段命名时，优先复用项目规则
- 模板用于约束结构，不用于替代正文和设定证据

## 结构稳定性

- 同一项目里，同一概念优先使用同一字段名
- 数组和对象层级保持固定，减少一章一改结构
- 字段扩展优先追加，不随意重命名已有键

## 待确认状态

如果文本里只有强推断，没有明确落锤：

- 可以写入 `pending_updates`
- 或在对应对象下标 `confidence: "inferred"`

## 与 Markdown 的关系

- Markdown 负责可读的叙事资产
- JSON 负责可校验的动态状态
- 两者冲突时，优先回查正文和已确认 canon
