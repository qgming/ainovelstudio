# 全自动写小说

这是一个面向长篇连载创作的内置工作流模板，重点是把章节推进和工作区文件维护绑在一起。

## 适用场景

- 已经有基础设定和章节目录，需要持续日更
- 希望把章节规划、创作、复核、设定维护串成稳定流水线
- 希望每轮循环都自动更新资料文件，而不是只产出正文
- 希望代理主动创建、更新、维护、合并或删除工作区文件，例如新人物出现时自动创建人物资料文件

## 默认流程

1. `start` 程序节点初始化本轮运行状态
2. 内置 `scene-plan` 代理读取工作区并创建或更新本章规划文件
3. 内置 `chapter-write` 代理基于工作区事实创作正文，并同步维护与本章相关的资料文件
4. 内置 `novel-review` 代理在判断节点中完成章节审查，并通过 `workflow_decision` 工具提交结构化结果
5. 判断未通过时，直接把问题反馈给章节写作节点修订当前章节；通过时进入设定更新
6. 内置 `snowflake-fiction` 代理维护人物、背景、线索、时间线与项目追踪资料
7. 结束节点根据最大轮次决定是否回到开始节点继续下一轮

## 运行语义

- 工作区文件是长期事实源，所有代理都必须先读工作区再行动
- 节点的主要职责是维护工作区文件，而不是追求某种固定输出格式
- 若出现新人物、新地点、新组织、新道具、新规则或新线索，应优先考虑创建对应资料文件
- 若发现重复、过期或错误文件，应在迁移有效信息后合并、重命名或删除
- 判断步骤通过 `workflow_decision` 工具传递 `pass`、`issues`、`revision_brief`
- 同一章节修订时，写作节点只能修订当前章节，不能误写下一章
- 审查失败后会直接回到章节写作节点，直到当前章节通过审查
- 当前章节通过判断并完成设定更新后，工作流才会进入下一章

## 绑定建议

- 先绑定一本已经初始化好的书籍工作区
- 确保工作区中已有完整大纲、人物设定和基础追踪文件
- 首次运行前建议确认内置代理资源已经初始化完成，尤其是 `scene-plan`、`chapter-write`、`novel-review` 与 `snowflake-fiction`

## 节点侧重点

- `scene-plan` 节点优先创建或更新章节规划、节拍、风险和连续性相关文件
- `chapter-write` 节点优先写正文，并同步补齐与本章直接相关的资料文件
- 判断节点正文只需简短说明，程序分支以 `workflow_decision` 工具结果为准
- `update-lore` 节点优先维护人物、地点、组织、线索、时间线、关系和状态文件
- 各节点都以“先读文件，再改文件，最后简短汇报”为默认执行方式

## 其他说明

- 所有节点都不要求严格输出 JSON
- 工作流本身不自动绑定书籍，进入详情页后由用户手动绑定
- 左侧基础区可直接设置最大循环次数，并支持切换为无限
- 运行历史和步骤复盘保存在数据库中，不写回模板文件
